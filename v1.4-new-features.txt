Traveller World & System Generator — What's New in v1.4.0
==========================================================

GEN-UI — DARK MODE
-------------------
The app now has a dark mode. Toggle it via View > Dark Mode in the menu bar. The
setting is saved and restored the next time the app starts.

- Native Qt controls (window background, group boxes, labels, and badges) switch
  to a dark palette.
- The world and system HTML cards also switch to dark colours, matching the app
  rather than following the operating system setting.
- Toggling while a result is displayed updates the card immediately — no need to
  regenerate.

BIOLOGIC ATMOSPHERE TAINT (WBH p.83)
--------------------------------------
- Tainted atmospheres (codes 2, 4, 7, 9) can now produce a Biologic taint
  subtype. This taint appears in the atmosphere detail card as a taint with
  subtype "Biologic" (code B), representing alien micro-organisms or spores
  suspended in the air.

- When a biologic taint is present, the world is guaranteed to have at least
  Biomass 1 — life must exist to produce the taint. This enforcement happens
  automatically and is consistent with WBH p.83.

- Previously, biologic taint rolls were silently rerolled because the native
  life system (biomass ratings) was not yet implemented. Now that biomass
  generation is in place, biologic taints are fully enabled.

BELT MEAN TEMPERATURE (NEW IN WORLD CARD)
------------------------------------------
- Asteroid belt mainworlds (Size 0) now display a Mean temperature in Kelvin
  in the World Body detail card. The temperature is computed using the same
  WBH p.47 formula as for planetary worlds, with atmosphere DM set to zero
  (belts have no retained atmosphere).

GEN-UI — SYSTEM DETAIL CONTROLS REDESIGN
-----------------------------------------
- The flat row of radio buttons and checkboxes for system-detail sub-options
  has been replaced by a "System detail" group box (QGroupBox). Enabling the
  group-box checkbox activates the group; disabling it automatically grays out
  all four child options (NHZ Atmospheres, Oxygen requires biomass, Advanced
  temperature, and Runaway greenhouse). The dependency between System detail
  and its sub-options is now immediately visible without having to try each
  option to discover which ones are linked.

GEN-UI — TRAVELLERMAP PANEL SHOW/HIDE
----------------------------------------
- When "Procedural" mode is selected, the Sector, Name, and Hex fields are now
  hidden entirely rather than displayed grayed out. They reappear when
  "TravellerMap" is selected. This removes visual clutter and eliminates the
  confusion of visible-but-inactive fields.

GEN-UI — TRAVELLERMAP LOOKUP FEEDBACK
----------------------------------------
- TravellerMap lookups now run on a background thread (QThread). As soon as
  Generate is clicked, the status line shows "Looking up <world> in <sector>…"
  and the Generate button is disabled until the response arrives. The UI
  remains responsive and interactive during the lookup. Previously the app
  would freeze for the duration of a slow or timing-out TravellerMap request.

ORBIT TABLE — HORIZONTAL SCROLL
---------------------------------
- The system orbit survey table (11 columns: UWP, Name, Orbit#, Ecc/Incl,
  Period, Notes, and others) now scrolls horizontally when the panel is
  narrower than its content. Column headers no longer wrap to multiple lines.
  This is particularly noticeable on narrower window sizes or when the Notes
  column contains long text.

STANDALONE BINARIES — SMALLER FILE SIZE
-----------------------------------------
- v1.4.0 binaries are substantially smaller than v1.3.x through a combination
  of:
    • 37 unused Qt modules excluded from the bundle
    • Qt translation files removed (100+ languages not needed)
    • Unused image format plugins removed (only SVG, JPEG, GIF, ICNS, ICO
      retained)
    • Python bytecode optimised (docstrings and assertions stripped)
    • Debug symbols stripped on Linux and macOS
    • UPX compression enabled on all platforms

- macOS builds are now provided as separate architecture-native binaries
  (Apple Silicon arm64 only from v1.4.0 onwards). The Intel x86_64 build has
  been discontinued.

OPEN SYSTEM JSON — SECONDARY WORLD DETAIL NOW RESTORED
--------------------------------------------------------
- When a system JSON is opened (File > Open JSON…), secondary world profiles
  are now fully restored from the saved file. Previously only the main system
  structure (stars, orbit slots, mainworld) was reconstructed; the detail block
  for each secondary world slot (SAH, social data, moons, moon orbits, belt/world
  physical) was silently discarded.

- The system card now shows complete secondary world information — the same
  content that appears immediately after using "Full detail" generation — when
  a file saved with full detail is reopened.

SYSTEM MAP — WORLD MARKER POSITION
------------------------------------
- World glyphs on the orbit arc diagram now appear one third of the way down
  the arc from the top endpoint, rather than at the very top. Markers no longer
  cluster at the tip of short arcs and sit closer to the visual centre of the
  diagram, making multi-world systems easier to read at a glance.

HABITABILITY RATING (WBH p.131)
---------------------------------
- Every mainworld now receives a Habitability Rating — a score from 0 to 10+
  that indicates how suitable the world is for unprotected human habitation
  without survival gear.

- The rating is deterministic (no dice roll): it starts at 10 and applies
  modifiers for world size, atmosphere code, hydrographics, solar tidal lock,
  surface temperature, and surface gravity.

- Temperature modifiers use the physics-based advanced temperature figures when
  Advanced temperature is enabled; they fall back to the basic temperature
  category (Frozen, Cold, Hot, Boiling) otherwise.

- The rating appears in a new Habitability section of the world card, with a
  descriptive label:

    0         Actively hostile world
    1–2       Barely habitable world
    3–5       Marginally survivable world
    6–7       Regionally habitable world
    8–9       Suitable for human habitation
    10 or higher  Terra-equivalent garden world

- Terra itself scores exactly 10. A world with a thin oxygen atmosphere, mild
  temperature, and Earthlike gravity might score 8 or 9.

SYSTEM CARD — HABITABLE ZONE AND MINIMUM ARMISTICE ORBIT COLUMNS
------------------------------------------------------------------
- The Stars table in the System tab now shows two additional columns for each
  star: MAO (Minimum Armistice Orbit) and HZ Orbit# range.
- MAO is the minimum safe orbit for a treaty world — roughly one-third of the
  distance to the next star. It is shown as an Orbit# value to 2 decimal places.
  The primary star has no MAO and shows a dash.
- HZ Orbit# shows the inner edge, centre, and outer edge of the star's Habitable
  Zone as three Orbit# values separated by dashes, e.g. "4.50 – 5.00 – 5.50".
  Stars without a computed Habitable Zone show a dash.

SYSTEM TAB — MAINWORLD DETAIL REMOVED
--------------------------------------
- The System tab no longer duplicates the mainworld's UWP statistics,
  atmosphere, hydrographic, biological, habitability, and notes sections.
  Those details appear only on the Mainworld tab, which shows them via the
  full world card. The System tab now shows stellar data and the orbital
  survey table only.

SEISMIC HEATING — ABSORBED INTO TEMPERATURE DISPLAY
------------------------------------------------------
- The former "Seismic temperature" row in the world card has been removed.
  Tidal/seismic heating is now folded directly into the Advanced mean
  temperature, High temperature, and Low temperature values shown on the card.
  If seismic heating raises any of those figures, the new higher value is what
  appears — there is no longer a separate "what if you add seismic heating"
  display field.

GAS GIANT MASS — WBH TABLE NOW USED
-------------------------------------
- Gas giant mass is now rolled using the correct WBH three-tier mass table,
  rather than being estimated from the gas giant's size category:

    Small gas giant (GS):  10–35 Earth masses    (5 × 1D+1)
    Medium gas giant (GM): 40–340 Earth masses   (20 × 3D−1)
    Large gas giant (GL):  350–3,300 Earth masses (D3×50 × 3D+4)

- This rolled mass is stored in the system JSON and used everywhere a gas giant
  mass is needed: moon generation, tidal stress calculations, and Roche limit
  checks. A fallback based on diameter is used when loading older JSON files
  that predate this change.

MOONS NEAR THE ROCHE LIMIT NOW CONVERTED TO RINGS
---------------------------------------------------
- If a significant moon's closest approach to its parent gas giant (accounting
  for orbital eccentricity) falls inside the Roche limit (2.0 planetary diameters),
  it is tidally disrupted and converted to ring material. The ring count on the
  gas giant increases by one. This check runs after eccentricity rolls, so only
  moons with genuinely unsafe orbits are disrupted.

GEN-UI — ONBOARDING CARD
-------------------------
- When the app starts and no world has been generated yet, the result area
  now shows a styled card explaining the three-step workflow:
  "Enter a name → Choose options → Click Generate". It also includes a
  keyboard shortcut hint ("Press Return in any field to generate") and a
  note about TravellerMap mode. Previously this area showed only a dim grey
  placeholder label.

ORBIT TABLE — MOON ECCENTRICITY AND INCLINATION
-------------------------------------------------
- The e/i column in the system card orbit table now shows eccentricity and
  inclination values for non-ring moons. For example, a moon with eccentricity
  0.042 and inclination 12.3° will show "0.042/12.3°" in its sub-row.
  Previously this column was always blank for moon sub-rows.
- Only moons with at least one non-zero value are shown; rings always show blank.

ORBIT TABLE — GAS GIANT MASS AND DENSITY IN NOTES
---------------------------------------------------
- The Notes column in the system card orbit table now shows the mass and
  computed density of each gas giant — for example "120 M⊕ · 1.34 g/cm³".
  Mass is the value rolled from the WBH three-tier table; density is derived
  from the mass and the size-code diameter.

RESOURCE RATING (WBH p.131)
-----------------------------
- Terrestrial worlds (Size 1+) now receive a Resource Rating in the World Body
  section of the world card. The rating runs from 2 to 12 and reflects how
  rich the world is in natural resources:

  2–4   Very scarce   (low density or low size)
  5–8   Typical       (mid-range density and size)
  9–12  Very rich     (high density and large size, or rich biosphere)

- The base value is computed from a 2D roll modified by world size and
  density (heavy iron-core worlds roll higher; icy low-density worlds roll
  lower). If the world has native life, biological modifiers (biomass,
  biodiversity, compatibility) are applied on top.

- The rating appears after Escape Velocity in the World Body detail section
  when World Physical detail has been generated.

SECONDARY WORLD CLASSIFICATIONS (WBH p.163)
--------------------------------------------
Inhabited secondary worlds and moons in a star system now receive a classification
that describes the role they play in the wider system. There are seven possible
classifications:

- Colony (Cy) — a populous world under colonial government
- Farming (Fa) — an agricultural world in the star's habitable zone
- Freeport (Fp) — an independently governed trade hub
- Military Base (Mb) — a garrison maintained by an authoritarian mainworld
- Mining Facility (Mi) — an industrial extraction site
- Penal Colony (Pe) — a prison world under a strict mainworld
- Research Base (Rb) — a scientific installation funded by a developed mainworld

Each classification has eligibility requirements based on the world's own
characteristics and the mainworld's profile. Colony and Farming are assigned
automatically when requirements are met; the others require a dice roll. Only the
first qualifying classification in the above order is assigned — a world has at
most one role.

The two-letter classification code (e.g. "Mi") appears alongside the world's other
trade codes in the system orbit table and in JSON output.

MAINWORLD SELECTION (WBH pp.155-156)
-------------------------------------
When generating a full star system, the generator now follows the World Builder's
Handbook procedure for choosing the mainworld. All terrestrial bodies in the system
are scored on four criteria and the highest-scoring world is chosen as the mainworld:

- Habitability rating (most important)
- Native sophonts present
- Resource rating
- Best refuelling access (gas giant satellite or liquid water)

Occasionally (on an unlikely three-dice roll of 18), the selection is made at
random from all candidates instead — representing the Referee's prerogative to
surprise even the algorithm.

If a secondary world outscores the orbital-generation candidate, it is promoted to
mainworld. Social characteristics (population, government, law level, starport,
and tech level) are then assigned to the selected mainworld and to all secondary
worlds relative to it.

This replaces the old behaviour where the mainworld was always whichever orbit
happened to be closest to the habitable zone centre.

SYSTEM MAP — 3D PERSPECTIVE RENDERING
--------------------------------------
The system map now has a perspective rendering mode (--perspective on the command
line; toggle via the System Map window in the app). In perspective mode:

- All orbits are shown as full 360° ellipses observed from 15° above the orbital
  plane, rotated 30° clockwise from above, giving the map a convincing 3-D look.
- Inclined orbits project a blurred shadow onto the orbital reference plane,
  and small angle symbols are drawn where the orbit crosses the reference plane.
- An isometric floor grid is drawn behind all orbits to reinforce the sense of depth.

SYSTEM MAP — ORBIT DEPTH CUES
-------------------------------
In perspective mode the near and far halves of each orbit arc are drawn at
different transparencies. The half that curves toward the viewer is fully opaque;
the half receding behind the orbital plane is drawn at 40% opacity. This makes it
immediately clear which part of the orbit is "in front" and which is "behind" —
the same technique used in scientific orbit diagrams.

SYSTEM MAP — 3D SPHERE GLYPHS AND RING SYSTEMS
------------------------------------------------
Stars, gas giants, and terrestrial worlds are now drawn as shaded spheres rather
than flat-coloured circles. Each sphere has a bright highlight in the upper-left,
the body's natural colour in the middle, and a darker shadow around the edge.

Gas giants that have a ring system in the generated data now display a
foreshortened ring annulus — a rear half drawn behind the sphere and a front
half drawn in front of it, so the planet appears to sit inside its rings.

SYSTEM MAP — DROP LINES FOR INCLINED BODIES
---------------------------------------------
In perspective mode, any world or companion star on an inclined orbit (displaced
above or below the orbital plane) now shows a short dotted line connecting the
bottom of its sphere to the orbital plane directly below it. This makes it easy
to read the body's height above or below the plane at a glance.

SYSTEM MAP — BELT PERSPECTIVE BANDS
-------------------------------------
Asteroid belt orbits are now rendered as filled annular bands — a filled ring
between the belt's inner and outer edge — instead of a coloured line or symbol.
In perspective mode the band is correctly foreshortened to match the viewing
angle. The band width is proportional to the logarithm of the belt's physical
extent in AU, matching the logarithmic scale used for orbital radii.

FASTAPI — TRAVELLERMAP SYSTEM MAP
----------------------------------
The system map shown in the FastAPI web UI now displays the correct number of
worlds, belts, and gas giants when a world is looked up via TravellerMap. The
map previously rolled a fresh procedural count rather than using the canonical
numbers from the TravellerMap database.

- In TravellerMap mode the system map is generated via a new
  `/api/map/system/svg` endpoint that honours the official PBG
  (Population / Belts / Gas-giants) counts from TravellerMap.
- Procedural mode is unchanged — a fresh random count is still rolled.
- The `system_map.py` command-line tool gains the same capability via the new
  `--sector` flag (e.g. `--sector "Spinward Marches" --name Regina`).

FASTAPI — MONGOOSE TRAVELLER THEME
------------------------------------
The FastAPI web interface has been restyled to match the Mongoose Traveller visual
identity.

- The page background is near-black. The accent colour is amber-orange (#c87828),
  matching the orange used throughout the Mongoose Traveller books and digital tools.
- The header has an amber bottom border and the title is rendered in uppercase with
  wider letter-spacing, giving it a military-document feel.
- Generate, Save, and active-tab highlights all use the amber accent.

FASTAPI — WORLD AND SYSTEM CARD COLOURS
-----------------------------------------
The HTML world and system cards have been restyled to match the Mongoose Traveller
theme.

- Light mode: warm parchment backgrounds (cream/tan). Section headers are amber-orange,
  uppercase, with subtle letter-spacing — similar to the style used in Traveller
  rulebook tables.
- Dark mode: warm near-black backgrounds instead of cold grey. Section headers use
  the same amber accent.

FASTAPI — MONGOOSE PUBLISHING COPYRIGHT NOTICE

A Mongoose Publishing copyright and fair use disclaimer now appears at the bottom of

FASTAPI — SECURITY IMPROVEMENTS
---------------------------------
Several security improvements have been made to the FastAPI web interface. These
are not visible during normal use but protect against misuse.

- All responses now include `Content-Security-Policy`, `X-Frame-Options`, and
  `X-Content-Type-Options` HTTP headers.
- The generation status bar no longer accepts unsafe HTML content — user-supplied
  text (such as a sector name) is handled as plain text only.
- The result iframes (World card, Mainworld card, System card) are sandboxed so
  their content cannot access the parent page's stored data.
- The system map image is verified to be a valid SVG before being displayed;
  invalid responses produce an error message rather than a broken image.
- Saved preferences (format, settlement type, source mode) are checked against a
  list of valid values before being restored.

BUG FIXES
---------
- Fixed: The mainworld card in the Full System page sometimes displayed as blank on
  the first generate. The iframe was made visible immediately (before the browser had
  finished parsing the card HTML), so the user saw only the empty iframe background.
  Switching to another tab and back forced a repaint and revealed the content. The card
  is now only made visible after its content has finished loading, so it always appears
  on first generate.

- Fixed: Worlds orbiting a gas giant as a satellite displayed seismic stress
  values millions of times larger than physically possible. The tidal stress
  formula (calibrated for AU-scale star-to-planet distances) was incorrectly
  applied at moon distances (~0.1–0.8 Mkm), amplifying results by a factor of
  10¹⁵ or more. The formula is no longer applied at moon distances; only the
  tidal amplitude contribution remains. Additionally, the Tidal Stress Factor
  is now capped at 500 to prevent overflow at the display layer.

- Fixed: Cold-temperature worlds with standard breathable atmospheres (vacuum,
  trace, thin, standard, dense, etc.) were incorrectly shown as having an
  Ammonia ocean instead of Water. Ammonia oceans only form in chemically exotic
  environments — worlds with exotic, corrosive, insidious, or unusual atmospheres
  (codes A–F). All other Cold worlds now correctly display Water as the surface fluid.

- Fixed: `has_gas_giant` flag was incorrectly re-derived from `gas_giant_count`
  when loading a world JSON. If the two values ever differed (e.g., a world
  generated before the count was tracked, or an edited JSON), the flag would
  be wrong. The saved boolean is now read directly; `gas_giant_count` is used
  only as a fallback for legacy files that predate the explicit field.

- Fixed: worlds at extreme orbital distances (very cold, near absolute zero)
  could display a mean temperature of 3 K due to unchecked formula extrapolation.
  The WBH footnote specifies that results below 10 K should be replaced with a
  random 6–11 K roll. This is now correctly implemented.

- Fixed (Windows binary): the System tab and any web-view panel showed a blank
  white page. QtWebEngineProcess.exe and the WebEngine resource files (.pak,
  locales) were not included in the PyInstaller bundle when PySide6-Addons is
  installed separately from the base PySide6 package. These files are now
  explicitly collected in traveller_gen_ui.spec.

- Fixed (Windows install): a ModuleNotFoundError for PySide6.QtWebEngineWidgets
  at startup. PySide6-Addons (which provides QtWebEngineWidgets) was missing
  from gen-ui/requirements.txt. It is now listed with a compatible version
  constraint alongside PySide6.

- Fixed: all eleven tech-level sub-categories in the Technology Profile now use
  the correct base TL, bounds, and dice modifiers from WBH §5. Previously each
  sub-TL was rolled from the world's High common TL with no DMs and approximate
  bounds; now each uses the base TL specified in the rules (e.g. Medical rolls
  from Electronics TL, not High TL) with the full DM table applied. Affected
  categories: Energy, Electronics, Manufacturing, Medical, Environmental, Land
  Transport, Sea Transport, Air Transport, Space Transport, Personal Military,
  and Heavy Military. The technology profile string displayed on the world card
  will differ from earlier versions for most worlds.

- Fixed: Sea Transport sub-TL was forced to 0 on worlds with no surface water
  (Hydrographics 0). The correct WBH rule applies a DM −2 penalty instead,
  representing minimal maritime capability — some dry worlds have subsurface
  oceans or exotic surface liquids.

- Fixed: Personal Military sub-TL was forced to 0 on worlds with Law Level 0
  (no restrictions on weapons). The correct WBH rule raises the lower bound to
  Manufacturing TL on such worlds, reflecting that weapons are freely available
  at manufacturing quality. A DM +2 is also applied, so low-law worlds tend to
  have better personal military hardware than moderate-law worlds.

- Fixed: Military sub-TLs now apply the correct government and law level DMs.
  Government 0 and Government 7 (Balkanised) worlds receive DM +2 to Personal
  Military TL. Governments 7, A, B, and F receive DM +2 to Heavy Military TL.
  Law Level 0 or D+ receives DM +2; Law Levels 1–4 or 9–C receive DM +1.
  These DMs were entirely absent in the previous implementation.
